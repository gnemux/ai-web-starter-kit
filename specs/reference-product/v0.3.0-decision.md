# XWLC v0.3.0 Decision

## Conclusion

Decision: **Go for the declared MVP3 scope**.

MVP3 passes its declared Reference Product and package-consumption scope. The
deployed CatCare journey, product data, owner/private-link boundary, anonymous
submission, AI/Billing/Credit return, Audit/Outbox enqueue, PostHog observation,
package upgrade, database rebuild, GitHub CI, and Vercel smoke have durable
evidence. No blocking defect remains for the MVP3 reference/staging/test scope.

This Go decision is not a production launch approval. The next product may
proceed to product-expansion judgment in GNE-274 and may start later on the
proven current stack, while production users, live AI, real payment, promised
notification delivery, or a Cloudflare/Hono runtime each retain explicit gates
below. Those gates do not invalidate the completed MVP3 scope.

## Cloudflare And Hono Adapter Readiness

Assessment: **Conditional**.

The reusable packages remain runtime-agnostic: they expose Actor/Session,
Owner and email-verification checks, safe capability context, generic share
gate states, DB scope/evidence shapes, and provider ports without importing
Next.js, Vercel, Supabase clients, Hono, or Cloudflare runtime objects. Auth is
therefore expressed as an Actor/Session boundary rather than a reusable-package
dependency on Supabase User, and email verification is a replaceable port.

The current application adapters still use Next.js/Vercel/Supabase/Postgres.
Cloudflare/Hono, Worker Auth, Email, D1/KV/Queues, and a non-RLS persistence
adapter are not implemented or executed. A future adapter must reproduce the
application-level actor/owner/resource/scope checks and the accepted negative
matrix because D1 does not supply the current Postgres RLS defense. This is a
future runtime gate, not an MVP3 blocker.

## Next Product Recommendation

GNE-274 completed the separate product-expansion judgment in
`product-expansion-decision.md`. The recommendation is **Conditional Go** for
generating a clean template candidate and creating the next approved product
repository from it on the reference/staging/test stack. The product must use
isolated routes, services, tables, specs, and tests while
consuming only public `@xwlc/*` contracts. It must not import CatCare product
code or treat CatCare tables, DTOs, pages, policies, or event unions as reusable
platform objects.

Provider-free Outbox state-machine and deterministic event-ID extraction stays
trigger-based: perform it only when a working second consumer would otherwise
copy the same CatCare semantics. MVP3 may produce a separately generated clean
template candidate now; promote it to multi-product-proven only after that
second-consumer comparison. The current packages are private workspaces, so a
cross-repository distribution/update channel remains `not_run` until an
independently maintained product needs shared updates.
